Kubeflow Fraud Detection Pipeline Architecture

Fraud detection ML workflow using Kubeflow Pipelines, Argo Workflows, Kubernetes/GKE, MySQL, ML Metadata, and SeaweedFS.

Architecture Diagram

Pipeline Steps

Step Purpose Output
load-data Creates an imbalanced synthetic fraud-like dataset. Raw dataset artifact
preprocess-data Reads the dataset and removes missing values. Processed dataset artifact
train-model Trains a RandomForestClassifier. Model artifact
evaluate-model Calculates model metrics. Accuracy, precision, recall, F1 score

Kubeflow Components

Component Role
ml-pipeline-ui Web UI for pipeline upload, experiments, and runs.
ml-pipeline Backend API for pipeline definitions, experiments, runs, and workflow creation.
workflow-controller Argo controller that launches pipeline tasks as Kubernetes pods.
mysql Stores pipeline, experiment, and run metadata.
metadata-grpc / metadata-envoy ML Metadata services for artifact lineage and execution metadata.
seaweedfs / minio-service S3-compatible artifact and log storage.

Runtime Flow

  1. Developer writes fraud_pipeline.py.
  2. The pipeline is compiled into fraud.yaml.
  3. fraud.yaml is uploaded into Kubeflow Pipelines UI.
  4. User creates or selects an experiment.
  5. User starts a one-off run.
  6. ml-pipeline creates an Argo Workflow.
  7. Argo launches one Kubernetes pod per pipeline step.
  8. Artifacts are written to SeaweedFS.
  9. Metadata is written to MySQL and ML Metadata.
  10. Metrics appear in the Kubeflow Pipelines UI.

Fraud Detection Metrics

For fraud detection, accuracy alone can be misleading because the data is imbalanced. The important metrics are:

  • precision
  • recall
  • f1_score
  • accuracy

Recall and F1 score are especially important when the goal is to catch rare fraud cases.
